# Links

Links come in two flavors: internal (other pages within your docs) and external (everything outside your docs domain). External links have enough nuance to warrant their own guidance below. Internal links are mostly straightforward, with one exception worth naming.

<Heading level={2}>
  The Plinko <sup>TM</sup> effect
</Heading>

When a page leans too heavily on internal links, readers bounce from page to page chasing context, like a Plinko disc ricocheting toward a slot that may or may not pay off. The fix isn't fewer links. It's better structure: content reuse to avoid scattering related information, progressive disclosure to surface detail only when needed, and focused procedures that don't demand a side quest to complete.

## Link text

Good link text works without surrounding context. Read just your links, skipping everything else on the page. If each one clearly describes where it goes, you're good. If not, rewrite them. Every link should be unique and meaningful. "Click here" tells readers nothing. Neither does a raw URL. If you must display a URL as the link text, drop the protocol and `www` prefix: [github.com/newrelic/docs-website](https://github.com/newrelic/docs-website) not [https://www.github.com/newrelic/docs-website](https://github.com/newrelic/docs-website). For external links, give readers a reason to click: not just where you're sending them, but why.

## External links

External links in documentation should open in new tabs to preserve user context, especially when users are mid-procedure or referencing supplementary information.

### Opening in New Tabs

The [W3C recommends](https://www.w3.org/WAI/WCAG22/Techniques/general/G200) opening new windows/tabs only when necessary. For documentation sites, this is the exception: users frequently navigate away mid-task to check a UI, read detailed specs, or explore related concepts. Opening external links in new tabs prevents context loss and allows users to easily return to their place in the documentation.

Many users have developed the habit of Cmd/Ctrl+clicking links to open them in new tabs, recognizing the value of preserving their reading context.

### Visual Indicators

W3C guidelines recommend warning users when a link opens in a new window/tab. The external link icon serves this purpose and is widely recognized across documentation sites.

**When to use the icon:**

- Standard inline external links
- Links within body text or procedures

**When to skip the icon:**

- Cards or large clickable areas that function as links
- Lists of external resources where it's obvious all items are external
- Cases where repetitive icons create visual noise

Provide manual overrides in your link component to handle these exceptions while maintaining the icon as the default behavior.
